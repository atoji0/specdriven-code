# 使い方

---

## 全体の流れ

Excel に業務要件を記載し、Copilot Chat に指示を出すだけで、業務仕様書 → 外部仕様書 → UI仕様書 → 実装定義書の順に成果物が生成されます。

```mermaid
flowchart LR
    A[業務要件\n（Excel + PNG）] --> B[業務仕様書]
    B --> C[外部仕様書]
    C --> D[UI仕様書\nデータ定義書]
    D -. 任意 .-> E[整合性チェック]
    D --> F[実装定義書]
    E --> F
```

---

## 1. 入力資料の準備

仕様書の品質はインプットの質で決まります。状況に応じて以下を参考に入力資料を用意してください。

### 1.1 状況別の準備方針

| 状況 | 基本方針 |
|------|---------|
| **現行システムあり** | As-Isの業務フロー＋課題リストをExcelに記述する。AIがTo-Beのフローを提案するので、それを叩き台に修正する |
| **現行業務あり・システム新規** | 現行の業務手順（紙・Excel管理の手順書等）＋課題・改善要望を記述する |
| **完全新規** | 「作りたいこと」「登場人物」「扱うデータ」の3点をExcelに記述する。細かい項目定義は後でよい |
| **既存システムのリプレース** | 現行画面のキャプチャ（PNG）＋現行フロー説明をインプットにする。現行仕様書があれば添付する |

### 1.2 Excelに書くべき内容

AIはExcelに記載された内容をもとに仕様書を生成します。最低限、以下の2点をテキストで記述してください。

- **現状の業務について**：業務の流れ・課題・改善要望、実施タイミング（日次・月次など）・利用頻度・データ量の目安、関係者（誰が何をするか）
- **システムでやりたいこと**：誰が何をしたいかを列挙する（例：「担当者が工事計画を一覧で確認・編集したい」「申請者がワークフローで申請・承認したい」）

以下を追加すると、生成される画面イメージの精度が上がります。

| 追加項目 | 内容 |
|---------|------|
| **サンプルデータ** | 実際の運用に近いサンプルを数行記載する。列名・値の具体性が画面項目の精度に直結する |
| **プルダウン候補** | 状態・種別・優先度などの選択肢一覧を記載する |
| **データ定義（テーブル名・主キー）** | テーブル名と主キーだけでも記載するとAIのドメイン判断が安定する。カラム詳細は後でよい |
| **用語集** | 業務固有の用語や略語があれば定義を記載する。フィールド名の精度が上がる |

> [!TIP]
> **完全新規の場合**は「登場人物（誰が使うか）」「扱うデータ（何を管理したいか）」「業務の流れ（大まかな手順）」の3点を書くだけでも仕様書の骨格が生成できます。

### 1.3 PNG・PDF の準備

AI は Excel のセルに書かれたテキストは読み取れますが、**オートシェイプ・SmartArt・シート内に貼り付けた画像はテキストとして読み取れません**。該当コンテンツはいずれかの方法で提供してください。

- **PNG キャプチャ**：該当シートをスクリーンショット（`[Win]+[Shift]+[S]`）して PNG ファイルとして配置する
- **PDF に変換して配置**：Excel ファイルをそのまま PDF として保存する。AI が自動的に PNG へ変換して読み取ります（追加ツール不要・Windows 標準 API 使用）

業務フロー図や設計書が **PDF 形式** で提供される場合も同様に、プロジェクトフォルダに置くだけで構いません。

## 2. プロジェクトの準備

`projects/` 配下にプロジェクトフォルダを作成し、Excel ファイルや PNG を配置します。（※グループフォルダ配下への配置も可）

```
projects/
└── <業務名>/
    ├── （任意のファイル名）.xlsx  ← Excel 入力資料
    ├── （任意のファイル名）.png   ← PNG キャプチャ（任意）
    └── （任意のファイル名）.pdf   ← PDF（任意。AI が自動で PNG 変換）
```

AI へ仕様書作成を依頼すると、配置した Excel と PNG は自動的に `入力資料/` フォルダに移動されます。

## 3. コマンド一覧

各ステップは以下のコマンドで実行します。

| ステップ | チャットに追加するもの | コマンド |
|---------|----------------------|---------|
| 業務仕様書を生成 | プロジェクトフォルダ（`<業務名>/`）（**Excel ファイルは直接添付不可**） | `仕様書を作成して` |
| 外部仕様書を生成 | `*_業務仕様書.md` | `続きをおねがい` |
| UI仕様書 ＋ データ定義書を生成 | `*_外部仕様書.md` | `続きをおねがい` |
| 成果品の整合性チェック（任意） | プロジェクトフォルダ | `成果品をチェックして` |
| 実装定義書を生成 | 対象の `*_UI仕様書_*.md` を VS Code で開く | `実装定義書を作成して` |
| HTML 変換 | プロジェクトフォルダ | `HTML変換して` |

## 4. 未確定事項（TBD）への対応

各ステップで仕様が未確定な箇所は **未確定事項（TBD）** として自動的にドキュメントの「未確定事項」章へ記録されます。

- TBD が残っている間は次のステップへ進まず停止します
- 追加要望・質問・相談がある場合も TBD に追記してください
- 仕様書の本文を直接書き換えても構いませんが、**AI は TBD をもとに判断するため、書き換えた内容は必ず TBD にも追記してください**（本文だけでは AI は認識できません）

TBD が発生した場合は、**該当ドキュメントの「未確定事項」章に回答を追記して保存**してから以下を実行します。

```
TBDに回答しました。続きをおねがい
```

### TBD の回答方法

「未確定事項」章の各項目には `【回答例】` が記載されています。`例` を削除し、必要に応じて回答内容を書き換えてください。

**修正前（未回答の状態）**

```markdown
## 未確定事項

- マスタメンテを行うロールが未確定。業務要件にロール記載なし。
  【回答例】製造ラインの作業者とする
```

**修正後（回答済みの状態）**

```markdown
## 未確定事項

- マスタメンテを行うロールが未確定。業務要件にロール記載なし。
  【回答】製造ラインの作業者とする
```

## 5. 仕様書のレビュー観点

AI が生成した仕様書は、TBD への回答だけでなく**内容の正確性・業務との一致**も確認してください。設計は以下の 3 STEP で段階的に固めていきます。修正が必要な場合は「未確定事項」章に追記してから `TBDに回答しました。続きをおねがい` を実行します。

> **STEP を戻らないために**：前の STEP で業務前提・画面分割方針を固めてから次に進んでください。STEP2 以降での業務フローの再定義や、STEP3 での画面設計の見直しは大きな手戻りになります。

| STEP | 確認レベル | 対象成果物 | このSTEPで固めること |
|:----:|-----------|-----------|---------------------|
| **STEP1** | 業務フローレベル | 業務仕様書 | 業務の流れ・役割・業務単位・画面分割の考え方 |
| **STEP2** | 個別画面操作レベル | 外部仕様書・UI仕様書 | 画面数・構成・画面遷移・操作可否・UIとしての成立 |
| **STEP3** | データ定義レベル | データ定義書 | データの型・桁・制約・削除制御・保持可能性 |

---

### 5.1 STEP1 ― 業務仕様書のレビュー

業務の前提ズレを防ぐことが目的です。**自動チェックしすぎると「正しそうな誤設計」が固まる**ため、業務担当者が実態を確認することが最重要です。

#### 業務フロー図 / UC構成図

- [ ] タスクの実行者（ロール）に誤りはないか
- [ ] 各タスクのデータ操作（参照・登録・更新・削除）は認識通りか
- [ ] 他システムからのデータ取得が存在する場合、取得元・方式は妥当か
- [ ] 例外系（差し戻し・取消・再実行）が業務的に存在するか
- [ ] 「1作業」と認識する業務単位で画面が定義されているか

---

### 5.2 STEP2 ― 外部仕様書・UI仕様書のレビュー

STEP1 で確定した業務前提・画面分割方針を前提として、**画面構成・操作が業務として成立しているか**を確認します。画面がないと指摘しにくいため、業務担当者も積極的に確認してください。

#### 外部仕様書

- [ ] 画面イメージ（ワイヤーフレーム）が想定通りのレイアウト・操作感になっているか
- [ ] 業務上必要なデータを画面から入力・変更できるか
- [ ] 画面で扱うデータの単位（1件 / 複数）に認識差がないか

#### UI仕様書

- [ ] ロールによって操作が変わる箇所が想定通りか（管理者のみ編集、など）
- [ ] ステータスによって操作できなくなる箇所が想定通りか（承認後は編集不可、など）
- [ ] 検索条件が実際の使い方と合っているか
- [ ] 入力項目に過不足がないか
- [ ] 入力方式が業務実態に合っているか（自由入力 / プルダウン / 別画面で検索して選択、など）

---

### 5.3 STEP3 ― データ定義書のレビュー

確定した画面仕様・操作仕様を前提に、**データとして正しく保持・運用できるか**を確認します。このフェーズはユーザーが想像しにくいため、STEP2 までに業務担当者との合意を取っておくことが重要です。

#### 外部仕様書（データ運用方針）

- [ ] データ削除時の制御方法（論理削除・物理削除）が想定通りか

---

## 6. HTML 変換

全成果品の Index ページを生成し、HTML 形式に一括変換します。

- 変換したいプロジェクトフォルダをチャットに追加した状態で実行します。

```
HTML変換して
```

変換完了後、`docs/<グループ>/<業務名>/index.html` をブラウザで開くと全成果品にアクセスできます。
