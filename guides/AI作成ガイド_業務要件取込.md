# 業務要件取込ガイド

## 目的

入力資料（Excel 等）の内容を Markdown に変換するためのガイド。**推測による補完・新規作成は禁止。**

本ガイドで行うこと：
- **資料の転記**：資料に記載された内容を意味を変えずに転記する。
- **業務フロー候補の分割抽出**：資料に含まれる業務手順・作業を時系列のグループ（フロー候補）に整理する。これが唯一 AI が整理してよい作業。
- **データ定義の材料変換**：テーブル定義・DDL・ER図を定型フォーマット（入力制約＋サンプル）に分解する。
- **固定値の転記**：選択肢・コードを箇条書きで整理する。
- **大きな表の分解**：認可権限表・グループ別項目定義表など、そのままでは Markdown で管理しにくい大きな表を分解する。

本ガイドで行ってはならないこと（→ **業務仕様書ガイドの責任**）：
- UC設計・フロー設計・CRUD補完
- 未確定事項の TBD 化
- **「未確定事項」章の作成**（出力テンプレートに存在しないため、絶対に追加しないこと）

## 入力資料

入力資料は以下の4種類。添付資料に明記された内容のみを転記し、AIが独自に補完・新規作成することは禁止。

| 入力資料 | 形式 | 必須 | 内容の例 |
|----------|------|:----:|---------| 
| Excel | xlsx | ○ | 業務要件・ユースケース記述、データ項目定義・テーブル定義、サンプルデータ、固定値（選択肢）リストなど、業務に関する様々な情報 |
| PNG画像 | png | - | ExcelのオートシェイプやSmartArtなどスクリプトで読み取れないコンテンツを、ユーザーが事前にキャプチャしたもの（業務フロー図・画面イメージ・ER図など）。PDFから変換した PNG も同様 |
| PDF | pdf | - | 業務フロー図・設計書・ER図などが PDF 形式で提供される場合。AI が自動的に PNG 変換する |
| DDL | sql | - | 既存システムのテーブル定義（既存システムの場合のみ） |

> **PNG/PDF はユーザーがプロジェクトフォルダ直下（`<業務名>/`）に配置し、チャットに添付すること。** Excel のオートシェイプ等はスクリプトでは読み取れないため、該当シートを PNG でキャプチャして提供する（`入力資料/` フォルダへの移動は AI が自動実行）。

### 初回の入力ファイル配置

ユーザーはプロジェクトフォルダ（`<業務名>/`）直下に入力素材を置くだけでよい。`入力資料/` フォルダの作成は不要で、AI が自動的に作成してファイルを移動する。

```
<業務名>/
├── （任意のファイル名）.xlsx   ← ここに置く（Excel 入力資料、任意の枚数）
├── （任意のファイル名）.png    ← ここに置く（PNG キャプチャ、任意の枚数）
├── （任意のファイル名）.pdf    ← ここに置く（PDF、任意の枚数）
└── （任意のファイル名）.sql    ← ここに置く（DDL、任意の枚数）
```

### PNG 画像の扱い方針

PNG の内容は**業務要件ドキュメントに転載しない**。リネーム後のファイル名と1行の概要のみを `## 入力資料` 章に記載し、画像はそのまま管理する。

### AI による入力資料の取込手順

以下の手順を**順に**実行してから業務要件 md ファイルを生成する。

1. **`入力資料/` を自動作成する**：プロジェクトフォルダ直下に xlsx、png、pdf、sql が存在する場合、`入力資料/` フォルダを作成してそこに移動する。Excel が開いたままの場合は移動をスキップし、ユーザーに手動での移動を依頼する。

2. **PDF を PNG に変換する**：`入力資料/` 内に PDF ファイルがある場合、以下のコマンドをファイルごとに実行する。追加ツール不要（Windows WinRT API 使用）。
   ```powershell
   .\scripts\Convert-PdfToImages.ps1 -PdfPath "<入力資料フォルダ内の PDF パス>"
   ```
   変換後の PNG (`<PDFファイル名>_p1.png`、`_p2.png`...) は同じフォルダに生成される。元の PDF はそのまま残す。

3. **`_extracted.md` を生成する**：以下のスクリプトを実行する。
   ```powershell
   .\scripts\Read-ExcelToMarkdown.ps1 -ExcelPath "<入力資料フォルダ内の Excel パス>"
   ```

4. **シートを分類し、取込方鍵を決める**：`_extracted.md` を読み、各シートを以下の種別に分類する。分類結果はチャットに一覧表示してからステップ5へ進む。

   | 種別 | 判定の目安 | 取込方針 |
   |------|-----------|---------|
   | **業務要件・フロー記述** | UC・業務手順・目的・ロールが文章で書かれている | `## 業務要件` に転記する |
   | **テーブル定義（DDL型）** | 列として「項目名・型・桁・制約」が定義されており、データ行ではなく定義行が並んでいる | `## データ定義（材料）` に入力制約形式に変換して取り込む |
   | **データサンプル（実データ型）** | 実際のレコードが行として並んでいる（テーブル定義列がない） | マスタ系 → `## データ定義（材料）` にサンプルとして取り込み、入力制約は AI が推定で作成（注記必須）。コード固定の参照表 → `## 固定値（材料）` に箇条書きで転記。レコードが多い場合は代表レコードのみ（3〜10件）に絞る |
   | **グループ別項目定義表** | 項目名が行、グループ（課・部署・ラインなど）が列になっており、セルに「○ / -」や「用いる / 用いない」が入っている | `## データ定義（材料）` の対象エンティティに `#### 表示制約` として追加する |
   | **帳票・印刷レイアウト** | セル結合が多い・印刷を前提とした**見た目優先**のシート（データが入っていてもレイアウト構成が主体） | `## データ定義（材料）` に**項目名のみ抽出**して取り込む（レイアウト・セル結合・書式は無視） |
   | **グラフ・図表・説明資料** | テキスト抽出が困難な図・画像メイン | スクリプトで読み取れた文字列のみ `## 業務要件` に転記。PNG 化を推奨としてチャットに記載 |

4. **PNG をリネームする**：まず `_extracted.md` を読んで Excel 内に PNG の説明があるか確認する。次に各 PNG （PDF から変換したものを含む）をチャットに添付して内容を確認する。Excel の説明と画像内容を照らし合わせて `<分類>_<説明>.png` 形式にリネームする（分類例：`業務フロー`・`画面`・ER図` など。内容に合わせて AI が判断する）。
5. **業務要件 md ファイルを生成する**：`_extracted.md` と変更後の PNG（チャットに添付）をもとに、ステップ4の取込方鍵に従って生成する。`## 入力資料` 章に PNG のファイル名と１行の概要を記載する。

## 成果物

- ファイル名：`＜想定される業務名＞_業務要件.md`
- 出力形式：Markdown
- 出力方法
  - 本文を画面表示せず、必ずファイルとして出力すること
  - 引用（cite…）をすべて削除した“クリーンな Markdown” として最終出力すること
  - 作成したmdファイルは、最後に、人間が見やすいように整形すること

## 記載ルール

### 業務要件

- 本章は、資料に記載された内容を**そのまま転記**する（推測による補完・新規作成は禁止）。
- 章立ては資料の内容に基づき自由に組み立てる（固定しない）。
- 原文の意味を変えずに転記する（文章化・体裁統一のための再構成は禁止）。
- 業務フロー候補の転記のみを行う（グループ化・UC導出は禁止。→ **業務仕様書ガイドの責任**）。
- 抽出が困難な場合は中断し、①中断理由、②不足している情報（追加で必要な資料・観点）を列挙する。

### 業務フロー候補

資料に含まれる業務フロー・業務手順をそのまま転記したもの。AIによるグループ化・UC導出はここでは行わない（→ 業務仕様書フェーズで実施）。

**業務フロー情報がない場合（テーブル定義のみ）は本セクションを省略する。**

#### 業務フロー・UC記述がある場合（PNG・Excelシート）

- フロー名・シナリオ名・シート名をそのまま `###` 見出しにする（リネーム・統合・分割しない）
- そのフロー/シナリオに登場するアクターとアクティビティを「**アクター**が〜する」形式でそのまま列挙する
  - PNG読み取り時：スイムレーン名をアクター、各ボックスのテキストをアクティビティとして転記する
  - 判読できない場合は `（判読不可）` と記載する
- ファイル入出力を伴う場合は末尾に括弧書きで明記する（例：`（Excel出力）`）
- 外部システム連携がある場合は末尾に括弧書きで明記する（例：`（○○システム連携）`）
- **グループ化・統合・分割・UC導出は行わない**

#### テーブル定義のみの場合

- `## 業務フロー候補` セクションは出力しない（業務フロー情報が存在しないため）
- 業務仕様書フェーズが `## データ定義（材料）` のテーブル定義を直接読んでグループ化・UC導出を行う

### データ定義（材料）

- 本章は **データ定義書を作るための材料整理** を目的とする。
- 各概念データは必ず「制約事項 → サンプル表」の順に表記する。
- テーブル定義やDDLが存在する場合、共通5項目（バージョン番号、作成者、作成日時、更新者、更新日時）は取り込み対象外とする。
- テーブル定義やDDLが存在せず、サンプルのみで作成した場合は、以下の文言を明記する。
  - ※注記：入力制約（種別、必須、長さ、制約）は、AIがExcelから推定で作成した資料であるため、UI設計前に必ず見直すこと。
- データ定義を行うための材料が **全く存在しない場合** は、何も出力しないこと。
  - ただし、利用者から指示がある場合のみ、**〇〇マスタ（仮）**として作成してもよい。

#### 入力制約
- 入力制約は Markdown 表で記載する。
- 列は固定：|項目名|種別|必須|長さ/桁|入力制限|備考|
- 種別：文字列/数値/選択/参照/日付/日時/フラグ/バイナリ
  - 選択：固定値リスト（選択項目）または固定の参照表が明記されている場合のみ。なければ推定せず文字列。
  - 参照：項目名や定義から他エンティティのキーを参照していると推定できる場合に使用する（例：`〇〇マスタキー`、`〇〇ID`）。DDL/テーブル定義でFKが明示されている場合も同様。推定できない数値は`数値`のまま。
  - バイナリ：画像・ファイル等のバイナリデータを保持する項目（例：写真、手書き画像）。長さ/桁は `-`。
- 必須：〇/-
  - 入力しなくても登録が成立する補助項目（添付/コメント/優先度/期限 等）は、-
  - 種別＝フラグの場合は、-
  - 上記以外は、〇
- 長さ/桁：N/-
  - 文字列（定義なしの場合）
    - 名称：100文字
    - 自由記述/備考：256文字
    - コード：サンプル最大長（サンプル無しは20文字）
  - 数字（定義なしの場合）
    - 整数：最大9桁
    - 金額：最大12桁
    - 小数あり：最大9桁・小数2桁
  - 種別＝選択の場合：-
  - 種別＝参照の場合：-
  - 上記以外は、-
- 入力制限：種別に従い、以下の内容を記載する。
  - 選択：**選択項目**参照 もしくは **参照表**参照
  - 参照：`→ 〇〇参照`（例：`→ Pilotコード参照`、`→ 費目参照`）
  - フラグ：値域のみ記載（0/1, Y/N等）
  - 文字列：サンプルデータから、以下の文字種を判定する。該当しない場合は、-とする。
    - 半角数字
    - 半角英数字
    - 半角英数字（大文字）
    - 半角英数字＋記号（サンプルに出たもの）
    - 桁数や書式が固定の場合は、文字種の後ろに括弧で付記する（例：半角数字（4桁固定））
  - 日付、日時、数値：-
- 備考：UIコントロールの動作に関する補足のみ記載する。なければ `-`（例：`オートコンプリート`）。

#### サンプルデータ
- サンプルデータは、制約事項の**下**に、Markdown表で記載する。
- サンプルのレコードが多い場合、**主要レコードのみ** に絞って貼る（目安：3〜10件）。
- サンプル内の表示ルール自体は決めないが、同一表内では表記を統一すること。
- 個人情報が含まれる場合は、必ず汎用的な名称に置換してから貼る。
- サンプルが存在しない場合、テーブル定義またはDDLからAIが想像して作成する。

#### 表示制約

- グループによって表示項目が異なるエンティティにのみ追加する（存在しない場合は本セクションを省略）。
- `入力制約`表とは独立したセクションとして設ける（入力制約表の内容を変更しない）。
- グループに分割せず、**全グループ・全項目を 1 本の表** にまとめる。
- 列見出しはグループの **略称（最大5文字程度）** を使い、表直前に凡例テーブルを記載する。

### 固定値（材料）

- 固定値とは、システム上でユーザーがメンテナンスしない、コード値が固定のものを指す。
- 固定値は、**記載例**に従い、箇条書きで記載する。
- 選択項目やテーブル定義の備考にて、定数の指定があれば、こちらに転記する。
- 固定値は添付資料に明記された値のみ転記する。AI側で独自に値を追加しないこと。
- `:` は「コード: 表示名」を表す区切りとしてのみ使用する。値のみの場合は `:` を利用しない。

**記載例**
- ＜固定値名＞
  - <値1>
  - <値2>

- ＜固定値名＞（コードが資料に明記されている場合のみ）
  - <コード1>: <表示名1>
  - <コード2>: <表示名2>

### 改訂履歴

- 改訂履歴は以下の列構成で、Markdown表で記載する。  
| 版数 | 改訂日 | 改訂者 | 改訂内容 |
- 改訂者は、セッション冒頭で以下のコマンドを一度だけ実行して取得したログインユーザ名を使用する。1回取得したら再実行不要。
  ```powershell
  $env:USERNAME
  ```
