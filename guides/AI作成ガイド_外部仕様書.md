# 外部仕様書 作成ガイド

## 目的

本書は、外部仕様書を **AIで安定して作成する**ための記載ルールおよび構成ルールを定義する。

## 本ガイドの前提

- 外部仕様書は、**システムの外部視点からの仕様（画面・バッチ・データ）**を整理し、関係者間の認識を揃えることを目的とする。

- **画面一覧・バッチ一覧は、業務仕様書（4章「業務の流れ」）内の各フローのUC構成図を正源泉とし、`rectangle` 名を画面名として使用する。** AIが独自に画面を追加・変更することは禁止する。
- 業務要件資料に記載された項目名・候補値・サンプルデータは画面イメージの参考として使用してよいが、項目の過不足・並び順の最終判断はUI仕様書策定フェーズとし、外部仕様書では「概念レベルの項目」として扱う。
- 画面操作手順・UI制御・実装方式など、設計／実装レベルの詳細には踏み込まない。
- 画面イメージは PlantUML の Salt 形式で表現する。

### 未確定事項(TBD)について

- 未確定事項は、**画面一覧または画面遷移図の構成が変わる**未確定の論点のみを列挙する。
  - 列挙する論点例：画面の追加・割愛が判断できないもの、画面遷移結路が変わるもの、データ一覧の構成（トランザクションの増減）が変わるもの
  - **列挙しない論点例**：入力項目の制約・絞り込みロジック・出力内容の詳細・部門列の確定など→「UI仕様書」で扱う
- 実装詳細（採番形式・入力制約・保持方式/連携方式等）は記載せず、論点レベルに止める。

## 入力資料

| 入力資料 | 必須 | 用途 |
|---------|:----:|------|
| `＜業務名＞_業務仕様書.md` | 〇 | **正源泉**。画面一覧・バッチ一覧はUC構成図の `rectangle` 名から生成する。画面種別・端末制約の判定にも使用する |
| `＜業務名＞_業務要件.md`（または `_extracted.md`） | - | **補助資料**。画面イメージの項目名・選択肢・サンプルデータの参考にする。データ一覧（概念）の種別・説明の補強にも使用する |
| `入力資料/` 内の PNG 画像 | - | **補助資料**（あれば）。画面イメージ画像は Salt 形式作成の参考に、業務フロー画像は画面確認に活用する |

## 成果物

- ファイル名：`＜業務名＞_外部仕様書.md`
- 出力形式：Markdown（PlantUML Salt ブロックを含む）
- 出力方法
  - 本文を画面表示せず、必ずファイルとして出力すること
  - 作成した md ファイルは、最後に人間が見やすいように整形すること
  - 仕様書生成後に、確認質問・提案を出力してはならない。不明点はすべて「未確定事項」として仕様書内に記載すること。

>
> | ファイル | 収録内容 |
> |---------|------|
> | `{業務名}_外部仕様書.md` | 1章（画面一覧）・2章（画面遷移図）・4章（バッチ一覧）・5～6章（データ）・7章（未確定事項）・8章（改訂履歴） |
> | `{業務名}_外部仕様書_{グループ名}.md` | 3章の画面イメージのうち、対象グループの画面のみ |
>
> - 分割単位は**機能グループ**（業務仕様書の業務フロー単位、または画面一覧の章構成）に従う
> - **画面一覧・画面遷移図は必ず親ファイルに残す**（Step 5でUI仕様書生成時の入力として使用するため）
> - 分割後、Step 5の実行時は親ファイルと全ての `{業務名}_外部仕様書_*.md` を合わせてインプットとする

## 記載ルール

### 画面一覧

- **業務仕様書4章内の各フローのUC構成図の `rectangle` 名をそのまま画面名として転記する。** UC構成図に存在しない画面をAIが独自に追加することは禁止する。
- **画面名・画面の粒度**に関するルールは `AI作成ガイド_業務仕様書.md` の5.5章（UC構成図）に定義されている。
- 以下の列構成でMarkdown表に整理する。  
  | No | 画面名 | 用途 | 画面種別 | 入力方式 | 対象データ（概念） |
- 用途には、何をする画面か（目的）を **1文（1行）** で記載する（手順・項目・ボタンは書かない）。
  - 後工程でブレやすいルールがある場合のみ、用途の末尾に括弧で追記してよい。
- 対象データ（概念）には、UC構成図の対応 `database` ノード名を転記する。
- **画面種別**は、UC構成図の矢印・コンポーネントから以下の順で判定する。上位が一致した時点で確定し、それ以降は判定しない。

  | 判定順 | UC構成図の条件 | 画面種別 |
  |:------:|--------------|----------|
  | 1 | UCから `component "WF基盤"` への矢印がある | Workflow |
  | 2 | UCから `artifact` への `<<print>>` 矢印がある | 帳票 |
  | 3 | UCから `database` への矢印が `<<read>>` のみ（`<<create>>`・`<<update>>`・`<<delete>>` なし） | 参照 |
  | 4 | **タブレット・モバイル専用画面**かつ入力あり（業務フロー・業務要件に端末制約の記載がある） | コンテキスト |
  | 5 | 上記いずれにも該当しない | 通常 |
  | - | マスタ選択ポップアップ（後述ルールにより追加） | 選択 |

- **入力方式**は、UC構成図の対応 `database` ノードへのCRUD矢印と、UCの処理内容から判定する。

  | 入力方式 | 定義 |
  |---------|------|
  | 基本 | 単票入力（通常） |
  | 一括 | 複数行を表形式でまとめて登録／更新する（親エンティティなし） |
  | 親子 | 親1件に対して子明細（1:N）を同一画面で管理する |
  | 親子選択 | 親1件の子行を既存データから選択して紐付ける |
  | 階層構造 | 多段階層データをツリー形式で登録・管理する |
  | - | 選択・参照のみ（入力なし）。帳票は常に「-」 |

  **「一括」と「親子」の判定基準（優先適用）**

  - UC が `<<read>>` する `database` が存在する → **親子**
  - `<<read>>` する `database` が存在しない → **一括**
  - `<<ref>>` は参照マスタ（プルダウン用）なので判定に使わない

- **マスタ参照の入力方式（選択画面の追加判断）**
  - デフォルト：プルダウン
  - コード・名称以外の属性（例：設備種別・所属組織・型式など）で絞り込みが有効になりそうな場合：ポップアップ検索画面（画面種別：選択）を画面一覧に追加
  - 追加した場合は必ず `## 未確定事項` にTBDとして記載してユーザーに確認する
  - 判断できない場合：プルダウンのままとする

### 画面遷移図

- 画面一覧に列挙したすべての画面を対象に、メニューからの遷移関係を Mermaid の `stateDiagram-v2` で表現する。
- **基本構造**：`direction LR` で左から右朕に配置する。
- **ノード名**：画面名をそのまま使用する。コンテキスト②（一覧）は `画面名一覧`、入力フォームは `画面名編集` と表現する。
- **矢印ラベル**：遷移の目的を簡潔に記載する。複数のトリガー（新規・選択）が同じ遷移先に行く場合は 1 本にまとめる。
  - 一覧 → 編集：`新規 / 選択`
  - 編集 → 一覧：`保存 / 削除`（削除なしの画面は `保存`）
  - メニュー → 画面：画面名のみ（ブラケット不要）

### 画面イメージ

- 画面イメージの一番最初に、以下の定型文を追加する。
  > ここに記載した画面イメージは、**暫定イメージ**です。UI仕様書を検討後、変更される可能性があります。
- 画面一覧に列挙した各画面について PlantUML の Salt 形式でイメージを作成する。
- 見出しは `### ＜画面名＞画面` とし、画面一覧の画面名に「画面」を付けた名称にする。
- 見出しの直下に、画面一覧の「用途」をプレーンテキストで記載する。
- 一覧・入力フォームの両方を含む画面は、`### ＜画面名＞画面` 内をさらに `#### 一覧画面` / `#### 入力フォーム画面` で区切る。コンテキスト画面など複数ステップがある場合は `#### 条件設定画面` / `#### 一覧表示画面` / `#### 入力フォーム画面` のように各ステップに「画面」を付ける。
- 単一構成（階層ツリー管理など）は `####` を設けず `### ＜画面名＞画面` 直下に Salt を記載する。
- WF画面（画面種別：Workflow）は後述の **Workflow** パターンに従って作成する。
- **グループ（課など）によって表示項目や構成が異なる場合は、代表1パターンのみ作成する。** 複数パターンのワイヤーはUI仕様書で作成する。代表パターン選定の判断はAIに委ねてよい。

#### Salt 記法のルール

- `@startsalt` ～ `@endsalt` で囲み、直後の行に `scale 1.2` を記述する。**`@endsalt` は必ず独立行**に記載し、次行に ` ``` ` を記載すること（同一行禁止）。
- 全体を `{+` / `}` で枠線ありのダイアログとして囲む。セパレータ（区切り線）は `--`。
- **ラベルと `:` の間にスペースを入れないこと**（正しい記法：`ラベル: | 値`）。
- **一覧テーブルの列**：外部仕様書では代表的な列（コード・名称・主要区分・状態等）を表示する。全列の確定はUI仕様書フェーズで行う。列数が多い場合は末尾列を `...` で省略し、省略した列名をイメージの直下に箇条書きで記載する。
- **入力フォームの項目**：省略せず、業務要件に定義されたすべての項目を表示する。項目が多い場合は `--` セパレータで意味のあるグループ（基本情報・詳細情報・フラグ類等）に分けて記載する。
- **画面タイトルの幅確保**：タイトルの直後に全角スペースのダミースペーサー（` 　　　　　　　　　　　　　　　　　`）を追記する。
- **テーブル内チェックボックスは `[X].` / `[ ].`**（`.` なしはボタンと解釈される。単体フィールドとして使う場合は `[X] 項目名` でよい）。
- **テーブル内の空セルには全角スペース（`　`）を必ず記入すること**（半角スペースのみ・空文字のセルはセル結合として扱われるため）。
- **画面名の命名規則**：`～マスタ管理` でなく `～管理`。入力フォームのタイトルは `～編集`（`～管理（入力フォーム）` 禁止）。
- **テキスト入力欄の全角スペースは必ず全角のみ使用すること**（半角スペースの混入でレイアウトが崩れる）。
- **入力フォームの列幅バランス**：`{ ラベル: | "値" }` ブロックは左列幅（最長ラベル）と右列幅（最長値フィールド）のバランスで描画される。最長ラベルが他の画面より長い場合（例：`測定具メーカー名:` = 8文字 vs 通常6文字）は、全フィールドの値スペースを2文字分ずつ増やして右列を広げること。
- **検索条件ブロックはフラットな構造にすること**：`{ { 条件 } { ボタン } }` のような二重ネストはレイアウト崩れの原因になる。`{ 条件 }` と `{ ボタン }` は並列に並べること。

**入力部品の Salt 記法一覧**

| データ種別 | Salt 記法 | 用途例 |
|----------|-----------|--------|
| テキスト（文字列） | `"　　　　　　"` | 名称・コード等 |
| テキスト（複数行） | `{SI }` | 備考・メモ欄等 |
| 数値 | `"999999"` | 数量・金額等 |
| 日付入力 | `"YYYY-MM-DD"` | 入力欄の日付 |
| 年月入力 | `"YYYY-MM"` | 年月のみ |
| ドロップダウン | `^選択肢^` | 区分・状態等 |
| チェックボックス（入力） | `[X] 項目名` / `[ ] 項目名` | 有効フラグ・応募フラグ等 |
| ボタン | `[ ボタン名 ]` | 検索・登録・削除・選択等 |
| 写真撮影（タブレット） | `[ 撮影 <&camera-slr> ]` | タブレット端末での画像添付。PCブラウザではファイル選択になるが、Salt上は `[ 撮影 <&camera-slr> ]` で統一する |
| テキスト表示（入力不可） | クォートなしでそのまま記載 | 自動生成値・ID表示等 |
| 日付表示 | `YYYY/MM/DD` とそのまま記載 | 一覧・詳細の日付表示 |
| フラグ表示（一覧） | `✓` / 空白 | 一覧テーブル内のフラグ |

#### 画面の基本構造

画面はすべて**タイトル**を持ち、以下の**部品**を積み上げて構成する。  
画面種別で「どの部品を並べるか」が決まり、入力方式で「入力形式」が加算される。  
Workflow のみ独立した固定構成を持つ（下記サンプル参照）。

**部品一覧**

- **タイトル**：全画面必須。右端にボタンを置く場合は `|` で区切る。
  - Excelアップロードボタンがある場合はタイトル行の右端に配置する（例：`{ <b>タイトル　... | [ 新規作成 ] | [ Excelアップロード ] }`）
- **検索条件**：条件フィールドの縦並び（`{}`）と検索ボタン行（`{ [ 検索 ] | [ クリア ] }`）を必ず別ブロックに分けて記述する（一つの `{}` にまとめない）。
  - ダウンロードボタンがある場合は `[ クリア ]` の後に `[ Excelダウンロード ]` / `[ CSVダウンロード ]` を追加する
- **一覧表示**：`{#` 〜 `}` テーブル。先頭列はリンク・`[ 選択 ]`・`[X].` のいずれか
- **サマリ表示**：`【項目名：値　項目名：値…】` 形式のテキスト行。コンテキスト②のタイトル直下に置き、設定済みの業務条件を表示する（例：`【作業日：YYYY/MM/DD　職場：〇〇職場　シフト：昼】`）
- **入力フォーム**：`ラベル: | 入力部品` の縦並び
- **一覧編集**：`<b>■ サブタイトル` → `{#` テーブル（行末 `[ 行削除 ]`）`}` → `[ 行追加 ]`
  - 明細種類が複数の場合は `--` で区切って繰り返す
- **編集アクション**：`{ [ 登録 ] | [ 更新 ] | [ 削除 ] }`

**マスタ画面の有効フラグ**

マスタ画面（入力方式：基本・階層構造）には有効フラグを必ず含める。業務仕様書・業務要件に削除フラグ・有効フラグ等に相当する項目が定義されている場合はその項目名を使用する。定義がない場合は `有効`（有効フラグ）として強制追加する。追加箇所：
- **検索条件**：`有効フラグ: | [X] 有効のみ` のチェックボックスをデフォルトONで追加する
- **一覧テーブル**：`有効` 列を末尾に追加し、サンプル値は `✓` で表示する
- **入力フォーム**：`有効フラグ: | [X] 有効` を末尾に追加する
- 上流で一部のみ定義されている場合は、未定義のデータに TBD を記載する

**画面種別**

1. **参照**：検索条件 → 一覧表示
2. **通常**：タイトル右に `[ 新規作成 ]` → 検索条件 → 一覧表示（コード列を `<U>コード` リンクで表現）。**入力方式＝一括の場合は一覧画面を作成しない**
   - Excelアップロードがある場合はタイトル右を `[ 新規作成 ] | [ Excelアップロード ]` とする
3. **単一選択**：タイトル右に `[ 閉じる ]` → 検索条件 → 一覧表示（先頭列 `[ 選択 ]`。ラジオボタン不使用）
4. **複数選択**：タイトル右に `[ 閉じる ]` → 検索条件 → 一覧表示（先頭列 `[X].` / `[ ].`）→ `[ 選択 ]`
5. **コンテキスト**：2画面構成
   - 画面1：入力フォーム（業務条件）→ `[ 一覧表示 ]`
   - 画面2：タイトル右に `[ 条件変更 ] [ 新規作成 ]` → サマリ表示 → 一覧表示（コード列リンク）。検索条件なし
6. **帳票**：ヘッダ情報 → データテーブル（テキスト表示のみ）

**入力方式（入力がある画面に加算）**

1. **基本**：入力フォーム → 編集アクション
2. **一括**：タイトル右に `[ 一括保存 ]` → 検索条件 → 一覧編集
3. **親子選択**：入力フォーム（親）→ 検索条件 → 一覧表示（先頭列 `[X].`）→ 編集アクション
4. **親子**：入力フォーム（親）→ 一覧編集 → 編集アクション
5. **階層構造**：タイトル右に `[ ノード追加 ]` → 検索条件（階層の各レベルに対応するプルダウン）→ ツリー（左ペイン）＋編集パネル（右ペイン）の2ペイン構成。**一覧画面は作成しない**
   - 右ペインには常に入力フォームと `{ [ 登録 ] | [ 更新 ] | [ 削除 ] }` を表示する

##### 部品 — Salt サンプル

各部品の Salt 記法フラグメント。実際の画面では `{+` 内に積み上げて使用する。

**タイトル**

~~~plantuml
@startsalt
scale 1.2
{ <b>画面タイトル　　　　　　　　　　　　　 | }
@endsalt
~~~

**検索条件**（条件フィールドとボタンは必ず別ブロックに分われて記述する。ダウンロードボタンがある場合は `[ クリア ]` の後に追加）

~~~plantuml
@startsalt
scale 1.2
{
  {
    条件A: | "　　　　　　　"
    条件B: | ^選択肢^
  }
  { [ 検索 ] | [ クリア ] }
}
@endsalt
~~~

**一覧表示**（先頭列はリンク・`[ 選択 ]`・`[X].` のいずれか）

~~~plantuml
@startsalt
scale 1.2
{#
コード | 名称 | 区分 | 更新日
<U>A001 | データA | A | 2026/02/01
<U>A002 | データB | B | 2026/02/02
}
@endsalt
~~~

**入力フォーム**

~~~plantuml
@startsalt
scale 1.2
{
  名称: | "　　　　　　　　　"
  区分: | ^選択肢^
  有効: | [X] 有効
}
@endsalt
~~~

### バッチ一覧

- **業務仕様書4章内のUC構成図に `<<batch>>` として記載された `rectangle` 名をそのままバッチ名として転記する。** バッチが存在しない場合は「- 特になし」と1行記載する。
- 以下の列構成で、Markdown表で整理する。  
  | No | バッチ名 | 種別 | 実行タイミング | 概要 |
- 種別には、定期実行／手動実行／イベント起動 を記載する。
- 概要には、何をするバッチ処理か（目的）を **1文** で記載する（実装方式・詳細条件は書かない）。

### データ一覧（概念）

- 業務で使用するデータを、以下の列構成で、Markdown表で整理する。  
  | No | データ名 | 種別 | 説明 |
- 種別には、トランザクション／マスタ／定数／外部システム／外部マスタ を記載する。
- 定数は「データ一覧（概念）」には列挙してよいが、候補値は記載しないこと。
- 説明には、「何のデータか」を1文で記載する。項目名や候補値（例：○○・○○等）を記載しない。
- データ一覧（概念）は、原則として正規化単位（エンティティ単位）で列挙する（親・子・孫を統合しない）。
- 「N:M関係」が存在する場合、中間エンティティ（「○○○関連」等）を必ずデータ一覧に追加する。
- WFを利用する際は、以下の２つのデータを保持する。
  - ○○申請（トランザクション） ... 申請対象（内容）を表すデータ。詳細な保持方式・項目・採番は本書では扱わない。
    - ただし、提示された資料内に申請情報（申請対象、状態等）が既に存在する場合、TBDにて採否を確認すること。
  - WF基盤（**外部システム**）... 現在の状態・状態遷移・担当者等を管理する。
- **他システムが管理するマスタを参照する場合**（例：POMS0のマスタを別業務アプリから参照する場合）、種別を「**外部マスタ**」として参照するテーブル単位で列挙し、説明に「〇〇が管理」を付記する。ER図にも「外部マスタ」種別として表現する。自システムで管理するマスタと明確に区別する。

#### データ運用方針

- 本節は**マスタ運用**と**WF申請対象データ運用**に関する記載のみを記載する。
- いずれの条件にも当てはまらない場合は、節の末尾に「- 特になし」と1行だけ記載する。
- **マスタ運用** ... 画面一覧に「マスタ管理（一覧/入力）」または「マスタ管理画面（マスタ運用産山」が存在する場合
  - 業務要件資料に「有効/無効」「廃止」「削除」等の運用が明記されている場合は、その記載を 最優先 で採用する。
  - 明記がない場合は、以下（有効フラグ運用）の 固定文 を出力する。
    - マスタの削除は、他データで参照（利用）されている場合は実行できない。
    - 参照（利用）されているマスタは、有効フラグを無効にして利用停止とする。
- **WF申請対象データ運用** ... 画面一覧に「WF申請」が存在し、かつ申請対象の「入力」画面が別に存在する場合
  - 以下の 固定文 を出力する。
    - ワークフロー申請対象のデータは、申請データと紐付けるため申請元データのIDを保持する。
    - 申請後の申請対象データは変更不可とし、差戻し時のみ変更可とする。

### データモデル（概念）

- データモデルは Mermaid の `erDiagram` で表現する。
- すべて日本語で表現する。
- **エンティティ名・関連のみを記載し、属性（列名）は記載しない。**列名・型・制約の詳細はデータ定義書（`＜業務名＞_データ定義書.md`）に委ねる。
- 物理テーブル設計・キー設計には踏み込まない。
- データ一覧（概念）の「トランザクション／マスタ／外部システム／外部マスタ」は、すべて本図に表現する。ただし、定数は除くこと。
- WFを利用する際は、WF基盤（外部システム）を一番最初に配置すること。
- エンティティの種別を色で区別する。`classDef` を使用し、以下の色を適用すること。

  | 種別 | fill | stroke |
  |------|------|--------|
  | マスタ | `#dbeafe` | `#3b82f6` |
  | トランザクション | `#ede9fe` | `#7c3aed` |
  | 外部システム | `#d1fae5` | `#059669` |
  | 外部マスタ | `#f0fdf4` | `#16a34a` |

### 未確定事項

- 業務要件に明記されていない仕様を推測・仮確定して記載した場合で、**画面一覧または画面遷移図の構成が変わる可能性があるもののみ**TBDとして本章に列挙する。画面構成に影響しない項目制約・表示ロジック等はTBDに列挙しない。
- 記載形式は箇条書きとし、論点ごとに1項目でまとめる。
- 各項目の直下に、AIが期待する【回答例】を1行で追記する。
- TBDが存在しない場合（すべての仕様が業務要件に明記されている場合）のみ、「特になし」と記載する。
