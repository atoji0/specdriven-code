# 業務仕様書 作成ガイド

## 目的

本書は、業務仕様書を **AIで安定して作成する**ための記載ルールおよび構成ルールを定義する。

- 本ガイドは「**どう動くか**」を設計するフェーズを担う。業務要件（「何があるか」）をインプットとして、フロー・UC構造・TBDを確定する。
- インプットとなる業務要件（UCの列挙）は、`AI作成ガイド_業務要件取込.md` に従って事前に整理されていること。

## 本ガイドの前提

- 業務仕様書は、**業務の骨格（関係者／業務フロー／ユースケース一覧）**を整理し、関係者間の認識を揃えることを目的とする。
- 画面一覧・画面イメージ・バッチ一覧・データ定義等の外部仕様は、別途「外部仕様書」に記載する。
- 画面操作手順・UI制御・実装方式など、設計／実装レベルの詳細には踏み込まない。

### 未確定事項(TBD)について

- 未確定事項は、業務骨格（状態・責務・結果）に影響する未確定の論点を漏れなく列挙する。
- ただし、項目列挙・候補値・採番形式・検索条件・入力制約・保持方式/連携方式などの詳細は記載せず、論点レベルに止める。

## 入力資料

| 入力資料 | ファイル名パターン | 必須 | 用途 |
|----------|-----------------|:----:|------|
| 業務要件 | `{business name}_業務要件.md` | 〇 | **正源泉**。フロー設計・UC構造の正源泉 |
| フロー・画面画像 | `入力資料/*.png` | - | 業務フロー・画面イメージの補足参照。画像内容のそのままの転載は禁止 |

## 成果物

- ファイル名：`＜業務名＞_業務仕様書.md`
- 出力形式：Markdown（PlantUML ブロックを含む）
- 出力方法
  - 本文を画面表示せず、必ずファイルとして出力すること
  - 作成した md ファイルは、最後に人間が見やすいように整形すること
  - 仕様書生成後に、確認質問・提案を出力してはならない。不明点はすべて「未確定事項」として仕様書内に記載すること。

## 記載ルール

### 目的

- 本書（業務仕様書）で **何を整理し、何に使うか**を簡潔に記載する。
- 章立ての説明や詳細背景は書きすぎない。

### 対象範囲

- 本業務で扱う範囲（ユースケース／機能の大枠）を箇条書きで整理する。
- 採番規則、除外条件、編集可否などの個別ルールは対象範囲に書かず、ユースケース一覧の概要 または 未確定事項 に記載する。
- **システム設計に直結しないが業務理解の共有に必要な情報**（業務の実施タイミング・利用頻度・データ量の目安など）も、入力資料に記載があれば対象範囲に含めて記載する。これらはシステム要件ではなく業務背景の共有が目的。

### 関係者（ロール）
- 業務上の役割を、以下の列構成で、Markdown表で整理する。  
  | 関係者（実在） | ロール（システム） | 主な責務 |
- **関係者（実在）**：現実の職種・部署・組織など、業務上の実在する主体を記載する。
- **ロール（システム）**：システム上の権限・役割を表す名称を記載する。業務フローのスイムレーンラベルと一致させる。
- **データ項目名（列名）をロール名として転用してはならない**（例：データ定義の「担当者」「主担当者」等の列名はロールではない）。
- 業務要件でロール名が「TBD」または未記載の場合は `＜TBD＞` とし、確定前に補完・推測しない。WFにより存在が示唐される主体は `＜TBD：小分類＞` で記載。
- WFで承認が複数段階でも、業務上の責務が統合できる場合は1つのロールにまとめてよい（例：`審査・承認者`）。

### 業務フロー

PlantUML のスイムレーン形式アクティビティ図で表現する。各フローの `#### 業務フロー` 直後に、そのフローの目的を1文で記述してから図に入ること。デバイス・端末制約など補足が必要な場合もこの1文に含める。

#### フロー分割の原則

業務要件の記載内容にあるアクティビティは削除・省略しない。

1. **マスタ管理と業務データは必ず別フローにする**（マスタ管理はドメインをまたいでも1フローにまとめてよい。マスタ管理フローでは複数UCを `group 要選択 { }` でまとめる）
2. **時系列で連続する複数の作業は1フローにまとめる**（システム外作業・日をまたぎでも、業務目的が同じなら同フロー）。ただし資料で別シナリオと明示されている場合は別フローにする
3. **時系列が連続しない作業は別フローにする**（前工程完了待ちで決まるタイミング差、担当者差が明確）

記述粒度は業務の複雑さに応じて判断する。業務フローは本来「誰が・何のために・何を達成したか」という業務アクション単位で記述する。ただし業務が単純でシステム操作そのものが業務となる場合（例：撮影・検索・削除しかない参照系アプリ等）は、UC番号を直接記載する操作レベルの記述を許容する。

#### 業務フロー図の書き方

| 対象 | ルール |
|------|--------|
| フロー名 | `### フロー名` 見出しには業務を表す簡潔な名称のみを使用する。項番は付与しないこと |
| スイムレーン | ラベルは関係者表の **ロール（システム）** と一致させる。ロールが切り替わるたびに `|ロール名|` を再宣言する |
| UC（システム処理） | `:UC番号:UC名; <<#D6EAF8>>` 形式。UC番号は全フローを通じた連番 |
| 手作業・業務行為（非UC） | `:アクティビティ名;` で記述。**体言止め**（例：`:申送り帳への記入;`）。システム内部操作（画面遷移・ステータス更新等）は書かない |
| `（補足）`付き名称 | `（` の直前に `\n` で改行する（例：`:UC3:異常処置記録入力\n（処置者追記）; <<#D6EAF8>>`） |
| note | 使用禁止。補足は `#### 業務フロー` 直後の目的文に含める |
| UC間アクティビティ（業務データフローのみ） | UC→UCの間に必ず業務的アクティビティを挿入する（トリガー・引継ぎ行為・判断確認）。手作業・紙記入・移動等も省略禁止。マスタ管理フローは不要 |
| 1レーンフローのスイムレーン強制表示 | スイムレーンが1つしかない場合、`stop` の後に改行して `\| \|` を追加し、強制表示する |

WFありフローの場合、以下のルールを追加で適用する。

| 対象 | ルール |
|------|--------|
| 業務フロー内容 | 入力→申請→承認→確認のハッピーパスのみ記載する。差し戻し・再申請・承認ルートの条件分岐は含めない（状態遷移図で表現） |
| WF承認側スイムレーン | 複数段でも1本にまとめてよい（例：`\|審査者 / 承認者\|`）。承認側UCは**「○○承認」の1つのみ**（「審査」名のUCを別途作成してはならない） |
| WF承認後のフロー | 承認完了・差戻しいずれも、承認側レーン内で`stop`せず、**申請者レーンに戻って業務アクティビティを記述してから`stop`** する（承認完了→`:承認完了の確認;`、差戻し→`:内容修正・再申請;`） |

#### 分岐・ループ

- 条件分岐は `if (条件？) then (ラベル) ... else (ラベル) ... endif`。
- **矢印クロス防止のため、ブランチの配置はレーンの移動方向で決める（必須）：**
  - **現在のレーンより右のレーンへ移動するブランチ → `else` 側**に配置する（矢印が左→右の自然な向きになる）
  - **現在のレーンと同一、または左のレーンへ移動・終了するブランチ → `then` 側**に配置する
  - 両ブランチが同じ方向（例：どちらも左レーンへ戻る）の場合は順序自由。処理が短い方を `then` に置く。
- 繰り返しは `repeat / repeat while` または `while ... endwhile`。

### 状態遷移図

WFを含むフローには、`#### 業務フロー` の直後（`#### UC構成図` の前）に `#### 状態遷移図` セクションを設け、PlantUMLの state ダイアグラムで表現する。図の前に、対象オブジェクトのステータス遷移の概要を1文で記載する。

#### WF基盤を利用する場合

**① 状態遷移図**と**② 承認ステップ詳細表**の2点を記載する。

**① 状態遷移図**

以下のルールに従う。業務フロー図に異なる記載があってもこのテンプレートに合わせる。合わない場合はTBDにする。
- `[*] --> 申請中` から開始（申請前の入力作業は状態ではないため含めない）
- `確認中`（申請者の承認完了確認）は必須。`承認中 --> 確認中 : 承認完了` `確認中 --> [*] : 完了`
- 差し戻し後は `再申請中`。再申請の遷移先は必ず最初の承認ステート（`申請中` に戻してはならない）
- `left to right direction` を先頭に必ず記述する

**記載例**

~~~plantuml
@startuml 状態遷移_WF_サンプル
left to right direction

申請中 : 申請者
承認中 : 承認者A → 承認者B
確認中 : 申請者
再申請中 : 申請者

[*] --> 申請中
申請中 --> 承認中 : 申請
承認中 --> 承認中 : 次ステップへ
承認中 --> 確認中 : 承認完了
確認中 --> [*] : 完了
承認中 --> 再申請中 : 差し戻し
再申請中 --> 承認中 : 再申請（ステップ1から）
再申請中 --> [*] : 取下げ
@enduml
~~~

**② 承認ステップ詳細表**

承認ステップごとの承認者・次ステップ・差し戻し先を一覧にする。単段承認（セルフループなし）の場合は省略する。

**記載例**

| ステップ | 承認者 | 次ステップ | 差し戻し先 |
|:-------:|--------|:--------:|----------|
| 1 | 承認者A | 2 | 再申請中（申請者） |
| 2 | 承認者B | 承認完了 | 再申請中（申請者） |

上記の基本例から、実際の業務に応じて以下の変更を加える。
- **単段承認**
  - `承認中` ステートのラベルを承認者1名に変更
  - セルフループ（`承認中 --> 承認中 : 次ステップへ`）を削除
  - `再申請中 --> 承認中` のラベルを `再申請` に変更（ステップ番号不要）
  - 承認ステップ詳細表は省略する

- **差戻し先が段によって異なる**
  - 承認ステップ詳細表の「差し戻し先」列に段ごとの差し戻し先を個別に記載

- **承認ルートが職場・グループごとに異なる**
  - `@startuml` ブロックはルート共通の**1つにまとめる**（最大段数に合わせた構造にする）
  - `承認中` ステートのラベルは「ルートにより異なる（表参照）」と記載する
  - 図の直後にルートごとの**承認ステップ詳細表**を追加する（表タイトルにルート名を付記）

- **上記以外の独自ルート**
  - ステートの追加・矢印ラベルの変更など基本例から逸脱した箇所を変更
  - 基本例から変更した理由を図の直前に1文で補足する

#### WF基盤を利用しない場合

状態遷移図の形式は自由。Excelや業務フロー図の内容に合わせて表現する。セルフループ（同一ステートへの遷移）があるステートのみ、図の直後に遷移詳細表を追加する。

### UC構成図

- 業務フロー（4章）と **1:1** で PlantUML のUCダイアグラムを1図作成する。業務フローが複数ある場合、フローごとに独立した `@startuml ... @enduml` ブロックに分けて記載する。
- **本図が外部仕様書の画面一覧・バッチ一覧の正源泉**となる。外部仕様書・UI仕様書はこの図の名称を参照して作成する。
- `@startuml` の直後に `left to right direction` を必ず記述する。これにより「アクター（左）→UC（中央）→データ（右）」の配置になる。

#### アクター

- `actor "ロール名"` で記載する。複数ロールが同一UCを操作する場合、`actor "ロール名A / ロール名B"` のように `/` で連結して1アクターにまとめてよい。
- **UCへの操作矢印がないロールはアクターに含めない**。業務フローのスイムレーンに登場するだけのロール（承認確認・参照のみ等）は対象外。スイムレーンのロールをそのままアクターに列挙してはならない。

#### rectangle（画面境界）

- `rectangle "画面名"` でUCをグループ化し、1画面を表す。
- **rectangle は操作単位ではなく業務画面単位でまとめる。** 検索・閲覧・削除など複数の操作UCが同一画面で行われる場合は1つの`rectangle`に統合する（操作ごとに別`rectangle`にしてはならない）。
- `rectangle` の名称が**外部仕様書の画面名の正源泉**となる。画面名・UC名は業務要件の `## 業務フロー候補` をもとに導出する（導出ルールは後述の **UC候補の導出** を参照）。業務要件にない画面・UCを独自に追加・変更してはならない。調整が必要な場合はフロー分割の原則に戻って再整理する。
- バッチ処理は `rectangle "○○バッチ" <<batch>>` として記載する。バッチが存在しない場合は記載しない。

WFがある場合、申請・承認は分割せず `rectangle "○○申請"` と1画面として記載する。`rectangle` 内のUCは「○○申請」（申請側UC）と「○○承認」（承認側の全ロールが操作，以下のUC候補の導出で追加）の**2つのみ**とする。

#### UC候補の導出

業務仕様書フェーズで `## 業務フロー候補` の転記内容をもとに、UC導出を行う。業務要件の記載内容にあるアクティビティは削除・省略しない。

**UC名・画面名**

- UC名・画面名ともに体言止めで自由に命名する
- WF申請画面の画面名は「○○申請」とする（UC構成図の`rectangle`名）
- WF承認側UCは「○○承認」とし、UC構成図のみに追加する（`rectangle`名には含めない）
- UC名は業務フロー内のUCアクティビティ（`:UC番号:UC名;`）と一致させる

**UC粒度（テーブル定義のみの場合）**

以下のルールは**同一ドメイン内のテーブル間にのみ適用する**。UC候補の統合は「**同一ドメイン + 各テーブルの項目種類が同種**」の場合のみ行う。

| パターン | UCのまとめ方 |
|---|---|
| 同一ドメイン ＋ 各テーブルが同種の項目構造（上位ID・コード・名称など同じ属性パターンが繰り返される階層） | **1UC**にまとめる |
| 同一ドメイン内で1:N:N以上（3段以上）、かつ段ごとに固有属性があり項目種類が異なる | 「親＋子」と「子＋孫」の2UCに分解する |
| N:M中間テーブルが複数ある（複数マスタへの対応設定） | 「対応設定管理」の1UCにまとめる |
| 別ドメインに属するテーブル（FKでつながっていても不可） | テーブルごとに独立したUCとする |

**UC粒度（業務フロー・UC記述がある場合）**

- **「データ入力・蓄積」と「WF申請」は必ず別UC・別画面にする（絶対ルール）**。タブレット等でレコードを入力する画面と、WFを起動する申請画面は必ず分離する。
- 同一画面で行われるCRUD操作（登録・更新・削除・検索）は分割せず**1UC**とする
- WFあり（申請→承認）：申請・承認で分割せず1UC（○○申請）とし、「○○承認」をUC構成図のみに追加する

#### 記号ルール

**記号の誤用は禁止**（例：データを `actor` で書く、外部システムを `database` で書くなど）。

**ノード**

| 要素 | 記号 | ルール・補足 |
|------|------|-------------|
| UCノード | `(UC番号: UC名)` | 番号・名称は業務フローのUCアクティビティ（`:UC番号:UC名; <<#D6EAF8>>`）と一致させる。対応する `rectangle` 内に配置する |
| データ（テーブル） | `database "○○"` | テーブル定義がある場合はプレフィックス除きのテーブル名をそのまま使用する（統合・別名付け禁止）。`rectangle` 名と同名の場合は `database "名前" as db_XXX` でエイリアスを付ける |
| 外部システム | `component "システム名"` | WFを含むフローには `component "WF基盤"` を必ず追加し、申請UCから `<<create>>`、承認UCから `<<update>>` の矢印を引く |
| ファイル | `artifact "ファイル名"` | ダウンロード・アップロード対象ファイル。外部システムが提供するファイルは `component "外部システム名" --> artifact : <<export>>` で提供元を明示する |

**矢印（操作ステレオタイプ）**

複数操作は `/` で連結する（例：`<<create/update>>`）。

| 操作 | ステレオタイプ | 対象 |
|------|:------------:|------|
| 登録 | `<<create>>` | `UC --> database / artifact` |
| 更新 | `<<update>>` | `UC --> database / artifact` |

## 生成後セルフチェック

業務仕様書ファイルを出力した後に、以下をすべて確認すること。1つでも該当する場合は即ファイルを修正すること。

### 業務フロー

- [ ] すべてのスイムレーンラベルが「関係者（ロール）」表のロール（システム）列と一致しているか
- [ ] 業務データフローで、UC→UCの間に業務的アクティビティが挿入されているか（マスタ管理フローは不要）
- [ ] 分岐で右レーンへ移動するブランチが `else` 側に配置されているか（矢印クロス防止）
- [ ] WFフローに差し戻し・条件分岐が含まれていないか（ハッピーパスのみ。差し戻しは状態遷移図で表現）
- [ ] WF承認後のフローが承認側レーンで `stop` せず申請者レーンに戻って終了しているか

### UC構成図

- [ ] UCへの矢印がないロールがアクターに含まれていないか（スイムレーンに登場するだけのロールを除外できているか）
- [ ] `database` / `component` / `artifact` の記号が正しく使い分けられているか
  - データ（テーブル）→ `database`、外部システム → `component`、ファイル → `artifact`
  - ロールや概念を `actor` や `database` で誤記していないか
- [ ] `<<ref>>` がマスタ管理フロー以外のUC構成図に使われていないか
- [ ] `rectangle` 名・UC名が業務フローのUCアクティビティ名と一致しているか
- [ ] WFありの場合、`component "WF基盤"` が追加されているか

### 整合性

- [ ] 業務フロー図とUC構成図がフロー数・構成（フロー名・UC名・rectangle名）で対応しているか
- [ ] 未確定事項に TBD が適切に列挙されているか（「常識的に決まる」として省略していないか）
